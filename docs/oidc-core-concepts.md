## 三个核心角色

OIDC 中涉及三个核心角色，理解它们的定义和关系是理解整个协议的基础。

### IdP（Identity Provider，身份提供者）

**定义**：负责验证用户身份、并向 SP 颁发凭证的中心化服务。

IdP 的职责：

1. **管理用户账号**：存储用户名、密码（或支持其他登录方式）、用户信息
2. **进行身份认证**：验证用户提交的凭据
3. **管理 SSO Session**：记录用户在 IdP 侧的登录状态
4. **颁发 Token**：生成并签名 ID Token、Access Token 等
5. **提供标准端点**：授权端点、Token 端点、UserInfo 端点等

**类比**：公安局。公安局负责验证公民身份并颁发身份证，任何机构（银行、酒店）都信任公安局颁发的证件。

**常见 IdP**：

- **公共 IdP**：Google、GitHub、微信、Apple ID
- **企业自建 IdP**：Keycloak、Auth0、Okta、本教程将构建的认证服务

### SP（Service Provider，服务提供者）

**定义**：依赖 IdP 进行用户认证的各个应用。SP自己不验证用户密码，而是把认证工作委托给 IdP。

SP 的职责：

1. **检测登录状态**：检查用户是否有本地 Session
2. **发起认证流程**：将未登录用户重定向到 IdP
3. **验证 Token**：验证 IdP 颁发的 ID Token 的签名和 Claims
4. **建立本地 Session**：验证通过后，在本地建立 Session
5. **处理本地授权**：决定已登录用户在本应用内能做什么

**类比**：需要查验身份证的各类机构（银行、酒店）。它们不自己签发身份证，而是信任公安局签发的证件。

### Client（客户端）

**定义**：发起 OIDC 授权请求的应用程序。这里的"客户端"指的是作为 OAuth2/OIDC 流程发起方的应用，通常是 SP 的一部分。

**重要澄清**：很多人会把"客户端"理解为"用户的浏览器"。实际上：

- **浏览器**：只是用户与 Client、IdP 交互的媒介
- **Client**：是 SP 的服务端代码，负责发起授权请求、处理 Token、维护 Session

```
┌─────────────────────────────────────────────────────────────┐
│                         用户浏览器                            │
│  （用户的代理，负责在用户、Client、IdP 之间传递指令和重定向）  │
└─────────────────────────────────────────────────────────────┘
              ▲                                 ▲
              │  授权请求、重定向                 │ 授权结果、重定向
              │                                 │
              ▼                                 ▼
┌─────────────────────┐              ┌─────────────────────┐
│   Client / SP       │              │   IdP               │
│  （发起授权请求）    │              │  （验证身份、颁发Token）│
│  - 验证 ID Token    │              │  - 管理用户账号       │
│  - 建立本地 Session  │              │  - 管理 SSO Session  │
│  - 处理业务逻辑      │              │  - 签名 ID Token     │
└─────────────────────┘              └─────────────────────┘
```

## 两种 Token：ID Token vs Access Token

这是 OIDC 中最容易混淆的概念。

### ID Token

**定义**：一个由 IdP 签名的 JWT，专门用于携带用户身份信息。

**用途**：告诉 Client"当前用户是谁"。

**特点**：
- 是一个 JWT，可以被 Client 用公钥自行验证（不需要每次都问 IdP）
- 包含用户的唯一标识（`sub`）和其他身份信息
- 有过期时间（通常较短，如 1 小时）
- `aud`（受众）字段必须是当前 Client 的 `client_id`，防止被其他应用冒用

**类比**：身份证。上面有你的照片（用户信息）和公安局的印章（IdP 签名），任何机构都可以自行验证证件真伪，不需要打电话给公安局。

### Access Token

**定义**：用于访问资源服务器的令牌。

**用途**：让 Client 能够调用 IdP 的 API（如 UserInfo API）。

**特点**：
- 通常是 opaque token（不透明，Client 无法自行解析，需要调用 IdP 的 API 验证）
- 有过期时间（通常比 ID Token 短）
- 用于代表用户访问第三方资源

**类比**：工作证明信。拿着这封信，你可以去档案室调取这个人的档案——但这封信不能证明这个人的身份。

### 两者的关键区别

| 特征 | ID Token | Access Token |
|------|----------|--------------|
| 格式 | JWT（可自验证） | 不透明或 JWT |
| 用途 | 证明"用户是谁" | 访问"资源/API" |
| 验证方式 | Client 用公钥自行验证 | 需要向 IdP 或资源服务器验证 |
| 包含信息 | 用户身份信息（sub、name、email） | 权限范围（scope） |
| 由谁使用 | Client（验证身份） | 资源服务器（授权访问） |

**重要记忆**：
- **ID Token** → **I** = **I**dentity，用于认证
- **Access Token** → 用于访问受保护资源（Authorization）

## Scope 体系

### 什么是 Scope

Scope 是 OAuth2/OIDC 中控制应用能获取哪些信息的机制。你可以把它理解为"用户给应用签发的权限清单"。

### 标准 OIDC Scope

OIDC 定义了几个标准 Scope：

| Scope | 含义 | 对应信息 |
|-------|------|----------|
| `openid` | **必须**。声明这是一个 OIDC 请求，而非普通 OAuth2 请求 | `sub` |
| `profile` | 访问用户的基本资料 | `name`, `given_name`, `family_name`, `picture`, `locale` |
| `email` | 访问用户邮箱 | `email`, `email_verified` |
| `phone` | 访问用户电话 | `phone_number`, `phone_number_verified` |
| `address` | 访问用户地址 | `address` |

### `openid` Scope 的特殊意义

`openid` Scope 是 OIDC 请求的**必须项**。如果请求中没有 `openid`，认证服务器会认为这是一个普通的 OAuth2 请求，不会返回 ID Token。

换句话说：

- **OAuth2 请求**：`scope=profile email`（没有 `openid`）
- **OIDC 请求**：`scope=openid profile email`（必须有 `openid`）

## 客户端类型：机密客户端 vs 公开客户端

### 机密客户端（Confidential Client）

能够安全保管 `client_secret` 的应用。

**特点**：
- 有服务端组件，可以安全存储 `client_secret`
- 授权码换 Token 时，必须提供 `client_secret`
- 可以使用 `client_secret_basic` 或 `client_secret_post` 认证方式

**示例**：有后端服务器的 Web 应用（BFF 模式）

### 公开客户端（Public Client）

无法安全保管 `client_secret` 的应用。

**特点**：
- 所有代码对用户可见，`client_secret` 无法隐藏
- 不能使用 `client_secret_basic` 或 `client_secret_post`
- 必须使用 PKCE（Proof Key for Code Exchange）增强安全性

**示例**：纯前端 SPA、移动 App

### 两种客户端的安全差异

| 维度 | 机密客户端 | 公开客户端 |
|------|-----------|-----------|
| client_secret | 可以安全存储 | 无法存储 |
| Token 换取 | 用 client_secret 验证身份 | 用 PKCE 验证身份 |
| 安全性 | 较高 | 需要额外保护（PKCE） |
| 适用场景 | 有后端服务的 Web 应用 | SPA、移动 App |

## IdP 与 SP 的信任关系

SP 如何信任 IdP 颁发的 Token？答案是**数字签名**。

### 信任机制

1. **密钥对**：IdP 持有一对密钥——私钥（绝密，用于签名）和公钥（公开，用于验证）
2. **Token 签名**：IdP 颁发 Token 时，用私钥对 Token 内容进行签名
3. **公钥分发**：SP 从 IdP 的 JWKS 端点获取公钥
4. **签名验证**：SP 收到 Token 后，用公钥验证签名是否有效

```
IdP（持有私钥）                           SP（持有公钥）
     │                                          │
     │  1. 用私钥签名 Token                     │
     ▼                                          │
┌─────────┐                                    │
│ Token   │                                    │
│ (含签名) │───────────────────────────────────>>>│
└─────────┘    2. 用公钥验证签名               │
                                              ▼
                                        ┌──────────┐
                                        │ 验证通过 │
                                        │ 信任用户 │
                                        └──────────┘
```

### JWKS 端点

公钥不是直接以文件形式发给 SP，而是通过一个标准端点获取：

```
/.well-known/jwks.json
```

返回格式：

```json
{
  "keys": [
    {
      "kty": "RSA",
      "use": "sig",
      "alg": "RS256",
      "kid": "2024-01",
      "n": "0vx7agoebGcQSuuPiLJXZptN9...",
      "e": "AQAB"
    }
  ]
}
```

SP 通过 `kid`（Key ID）找到对应的公钥进行验证。

## 本篇小结

本篇介绍了 OIDC 的核心概念：

1. **三个角色**：IdP（身份提供者）、SP（服务提供者）、Client（客户端应用）
2. **两种 Token**：ID Token（证明用户身份，可自验证）和 Access Token（访问资源，需要验证）
3. **Scope**：控制应用能获取哪些信息，`openid` 是 OIDC 请求的必须项
4. **客户端类型**：机密客户端（可持有 secret）和公开客户端（必须用 PKCE）
5. **信任机制**：通过数字签名和 JWKS 端点实现 IdP 与 SP 之间的信任传递

这些概念将在后续章节中反复出现，建议在继续学习之前确保对本篇内容有清晰的理解。
